# 04. Дорожная карта

Пошаговый план от пустого репозитория до полного портала. Формат: **MVP → полный**. Каждая фаза даёт **проверяемый результат** (Definition of Done), который можно задеплоить и показать.

> Принцип: сначала «тонкая вертикаль» (одно упражнение работает и задеплоено), потом наращиваем вширь. Так в любой момент есть рабочая версия.

```
Ф0 Каркас ─▶ Ф1 Ядро ─▶ Ф2 Базовые ─▶ Ф3 Тест-режим ─▶ Ф4 Контент ─▶ Ф5 Продвинутые+кастом ─▶ Ф6 Полировка
   деплой      1 упр.     5 типов        счёт+сводка       3 статьи        4 компонента            UX/моб.
```

---

## Фаза 0 — Каркас и деплой (фундамент)
**Цель:** пустой портал собирается и открывается на GitHub Pages.

Шаги:
1. `npm create vite@latest . -- --template react-ts` (в текущей папке).
2. Установить базовые зависимости: `react-router-dom`, `@mdx-js/rollup`, `@dnd-kit/core`, `@dnd-kit/sortable`, `lucide-react`.
3. `vite.config.ts`: `base: '/promt_engineering_academy/'`, подключить MDX-плагин.
4. Роутинг на `HashRouter`: маршруты `/` (Home), `/article/:slug`, `/about`.
5. Минимальный layout: `Header` с навигацией, контейнер контента, футер.
6. Заглушка главной: «Prompt Engineering Academy» + список-заглушка статей.
7. `.github/workflows/deploy.yml` (см. [`05-deployment.md`](./05-deployment.md)); включить Pages в настройках репозитория.
8. `.gitignore` (node_modules, dist), первый коммит, пуш в `main`.

**DoD:** по адресу `https://erzyukov.github.io/promt_engineering_academy/` открывается портал-заглушка, навигация между пустыми страницами работает.

## Фаза 1 — Ядро системы упражнений
**Цель:** один компонент проходит весь путь «данные → ответ → верно/неверно» в инлайн-режиме.

Шаги:
1. `core/types.ts` — `ExerciseStatus`, `ExerciseResult`, `ExerciseProps`.
2. `core/QuizContext.tsx` — контекст (пока возвращает `null` вне теста).
3. `core/useExercise.ts` — состояние `idle/answered/...`, `submit`, `reset`, связь с контекстом.
4. `core/validators.ts` — начать с `exactMatch`, `setMatch`.
5. `core/ExerciseShell.tsx` — обёртка: вопрос, слот, кнопка «Проверить», блок обратной связи (верно/неверно + объяснение).
6. Реализовать `MultipleChoice` (single) на этом фундаменте.
7. Тестовая MDX-страница со встроенным `<MultipleChoice/>`.

**DoD:** на тестовой странице `MultipleChoice` принимает ответ, показывает «верно/неверно» + объяснение, кнопка «Попробовать снова» сбрасывает состояние. Задеплоено.

## Фаза 2 — Базовые компоненты (5 типов)
**Цель:** весь базовый набор работает в инлайн-режиме.

Шаги:
1. Дополнить `MultipleChoice` режимом multi (+`setMatch` с частичным баллом).
2. `TrueFalse` — две кнопки, обязательное объяснение.
3. `FillTheBlank` — инпуты в шаблоне, нормализация ввода.
4. Подключить `@dnd-kit`; `MatchPairs` — drag-and-drop пар + `pairsMatch` (частичный балл) + клавиатурная альтернатива.
5. `OrderSteps` — сортировка списка (`@dnd-kit/sortable`) + `orderMatch` + кнопки ↑/↓.
6. Визуальное различение типов: иконка + акцентный цвет в `ExerciseShell`.
7. Демо-страница «витрина компонентов» со всеми пятью.

**DoD:** все 5 базовых типов работают инлайн, показывают корректную обратную связь, частичная правильность считается у `MatchPairs`/`OrderSteps`/multi-`MultipleChoice`. Витрина задеплоена.

---

## Фаза 3 — Тест-режим
**Цель:** последовательность упражнений с общим счётом, прогрессом и итогом.

Шаги:
1. `QuizProvider` — собирает `ExerciseResult` от вложенных упражнений через контекст.
2. `Quiz.tsx` — обёртка-последовательность: показывает вопросы (по одному или списком), хранит ответы.
3. `ProgressBar.tsx` — прогресс «отвечено N из M».
4. Агрегация баллов: `Σ(score·maxScore)/Σ(maxScore)` → процент.
5. `QuizSummary.tsx` — итог: процент, разбор по вопросам (верно/частично/неверно), рекомендации.
6. Проверить, что **те же** компоненты из Фазы 2 работают внутри `<Quiz>` без изменений (режим определяется контекстом).

**DoD:** демо-тест из 4–5 разнотипных упражнений: прогресс-бар заполняется, в конце — корректный процент и сводка с разбором. Один и тот же компонент работает и инлайн, и в тесте.

## Фаза 4 — Контент (3 статьи)
**Цель:** наполнить портал статьями со встроенной практикой.

Шаги:
1. `content/registry.ts` + рендер карточек на `Home`; роут `/article/:slug` грузит MDX.
2. **Статья №1** «Как работают LLM» целиком (см. [`03-content.md`](./03-content.md)): текст + 3–5 инлайн-тестов + финальный `<Quiz>`.
3. **Статья №3** «Галлюцинации»: текст + инлайн `TrueFalse`/`MultipleChoice` + финальный тест. (`SpotTheHallucination` подключим в Ф5 — пока заглушка/без него.)
4. **Статья №2** «Структура промпта»: текст + `MatchPairs` + финальный тест. (`PromptBuilder`/`CaseStudy`/`PromptRefinementLadder` — в Ф5.)
5. Оформление статьи: типографика, оглавление, время чтения.

**DoD:** 3 статьи открываются с главной, в каждой есть работающие инлайн-тесты и финальный тест с подсчётом. Задеплоено.

---

## Фаза 5 — Продвинутые + кастомный компонент
**Цель:** добавить «понимание и применение» поверх «запоминания».

Шаги:
1. `SpotTheHallucination` — клик по спанам, `setMatch` по галлюцинациям → встроить в статью №3.
2. `PromptBuilder` — палитра блоков + слоты (drag-and-drop) + превью структуры → статья №2.
3. `CaseStudy` — textarea + живой чеклист эвристик + сравнение с эталоном → статья №2.
4. ⭐ `PromptRefinementLadder` (кастомный) — пошаговое улучшение промпта с заготовленными «ответами AI» → статья №2.
5. Встроить продвинутые упражнения в финальные тесты, где уместно.

**DoD:** все 9 типов работают; продвинутые и кастомный встроены в статьи №2 и №3; кастомный компонент полностью функционален. Задеплоено.

---

## Фаза 6 — Полировка
**Цель:** довести до качества «можно показывать».

Шаги:
1. Сохранение прогресса в `localStorage` (пройденные тесты, набранные баллы) — хук `useLocalProgress`.
2. Адаптивная вёрстка: проверить мобильный (drag-and-drop → тач, либо клавиатурная/кнопочная альтернатива).
3. Доступность: фокус, `aria`-атрибуты, навигация с клавиатуры для всех упражнений.
4. Визуальный проход: единая палитра, иконки типов, состояния hover/focus, анимация обратной связи.
5. Страница `About` (о проекте, как устроен), мета-теги, favicon.
6. Финальная вычитка контента и проверка всех упражнений.

**DoD:** портал адаптивен, доступен с клавиатуры, прогресс сохраняется, контент вычитан. Соответствует Definition of Done из [`overview.md`](./overview.md).

## Зависимости между фазами
- Ф1 требует Ф0 (каркас + деплой-пайплайн).
- Ф2 строится на ядре Ф1.
- Ф3 требует ≥ 2 готовых компонентов из Ф2 для осмысленного теста.
- Ф4 требует Ф2 (инлайн) и Ф3 (финальные тесты).
- Ф5 требует Ф4 (есть статьи, куда встраивать).
- Ф6 — финальная, поверх всего.
