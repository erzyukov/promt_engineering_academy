# 02. Компоненты-упражнения

Все компоненты следуют единому контракту из [`01-architecture.md`](./01-architecture.md): принимают `data`, используют `useExercise`, рендерятся в `ExerciseShell`, возвращают `ExerciseResult`. Ниже — спецификация каждого.

## Базовые компоненты

### MultipleChoice — выбор ответа
- **Данные:** `question`, `options[]`, `correct[]` (1 id = single, ≥2 = multi), `explanation?`.
- **UI:** радиокнопки (single) или чекбоксы (multi). Тип определяется длиной `correct`.
- **Проверка:** single → `exactMatch`; multi → `setMatch` (доля верных без лишних).
- **Состояния:** выбранные после ответа подсвечиваются ✓/✗; правильные, что не выбрал, — подсветка-подсказка.

### TrueFalse — верно/неверно
- **Данные:** `statement`, `answer: boolean`, `explanation` (обязательно — суть упражнения в объяснении).
- **UI:** две кнопки «Верно» / «Неверно».
- **Проверка:** бинарная. После ответа всегда показываем `explanation`.

### FillTheBlank — заполни пропуск
- **Данные:** `template` с маркерами `{{0}}`, `blanks[]` (для каждого: `accept[]` — допустимые варианты), `explanation?`.
- **UI:** текст с `<input>` на месте пропусков; опционально — режим выбора из подсказок (chips).
- **Проверка:** нормализация (trim, нижний регистр, ё→е) + сверка с `accept[]`. Несколько пропусков → каждый бинарно, общий балл = доля.

### MatchPairs — соедини пары
- **Данные:** `left[]` (термины), `right[]` (определения), `correct: {leftId: rightId}`, `explanation?`.
- **UI:** два столбца; drag-and-drop правых на левые (через `@dnd-kit`). Альтернатива с клавиатуры: выбрать левый → выбрать правый.
- **Проверка:** `pairsMatch` — доля верно соединённых. 3 из 4 = `0.75` (статус `partial`).
- **Feedback:** верные связи — зелёные, неверные — красные, с указанием правильной пары.

### OrderSteps — расставь по порядку
- **Данные:** `steps[]` (в перемешанном виде), `correctOrder[]` (массив id), `explanation?`.
- **UI:** вертикальный список, перетаскивание для смены порядка (`@dnd-kit/sortable`). Клавиатура: кнопки ↑/↓ на каждом шаге.
- **Проверка:** `orderMatch` — доля шагов на своих позициях.

## Продвинутые компоненты

### SpotTheHallucination — найди галлюцинацию
- **Идея:** дан «ответ AI», нужно выделить фрагмент с выдумкой/ошибкой.
- **Данные:** `text`, `spans[]` (готовая разбивка на кликабельные фрагменты с `id`), `hallucinationIds[]`, `explanation`.
- **UI:** текст заранее разбит на спаны (предложения/фразы). Пользователь кликом отмечает подозрительные фрагменты — отмеченные подсвечиваются. (Разбивка на спаны вместо «свободного выделения мышью» делает проверку детерминированной и доступной.)
- **Проверка:** `setMatch` по выбранным спанам против `hallucinationIds`. Частичный балл, если галлюцинаций несколько.

### PromptBuilder — собери промпт из блоков
- **Идея:** собрать качественный промпт из блоков (Роль, Контекст, Задача, Формат, Ограничения).
- **Данные:** `blocks[]` (доступные кусочки, в т.ч. лишние/слабые), `slots[]` (Роль/Контекст/Задача/Формат), `rubric` (какой блок куда подходит), `explanation`.
- **UI:** палитра блоков + зоны-слоты, drag-and-drop. Справа — превью собранного промпта (визуализация структуры цветными секциями).
- **Проверка:** `checklistScore` — за каждый правильно заполненный слот балл; лишние/слабые блоки снижают. Частичный балл.

### CaseStudy — улучши плохой промпт (без LLM)
- **Идея:** дан слабый промпт, нужно его переписать. Оцениваем **эвристиками**, а не реальным AI.
- **Данные:** `weakPrompt`, `criteria[]` (каждый: `id`, `label`, `detect(text)→bool` — например «есть роль», «указан формат вывода», «есть конкретика/числа», «задан объём»), `referenceGood` (эталон для показа), `minScore`.
- **UI:** textarea с исходным промптом + чеклист критериев, который **подсвечивается в реальном времени** по мере правок. После «Проверить» — сравнение с `referenceGood`.
- **Проверка:** доля выполненных критериев (`checklistScore`). Это честная статическая оценка «улучшения»: не «правильно/неправильно», а «насколько промпт стал полнее».
- **Эвристики (примеры `detect`):** наличие role-маркеров («ты —», «выступи как»), указание формата («в виде списка/таблицы/JSON»), ограничения («не более», «только»), конкретика (числа, имена), контекст (длина/структура).

## Кастомный компонент

### PromptRefinementLadder — «Лестница улучшения промпта» ⭐
> Обязательный по ТЗ собственный тип. Идея напрямую учит главному навыку — итеративному улучшению промпта.

- **Идея:** дана расплывчатая задача и слабый стартовый промпт. На каждой «ступени» пользователь выбирает **одно** улучшение (добавить роль / контекст / формат / ограничение / пример). После каждого выбора обновляется заготовленный «ответ AI» — видно, как он становится лучше. Компонент визуализирует причинно-следственную связь «элемент промпта → качество ответа».
- **Данные:**
  ```ts
  interface RefinementLadderData {
    task: string;                 // что нужно получить от AI
    basePrompt: string;           // слабый старт
    rungs: {                      // доступные улучшения
      id: string; label: string;  // напр. «Добавить роль эксперта»
      addsTo: string;             // как меняется промпт
      element: 'role'|'context'|'format'|'constraint'|'example';
    }[];
    // заготовленные ответы, ключ = отсортированный набор применённых element'ов
    responses: Record<string, string>;
    optimalSet: string[];         // какой набор элементов считается лучшим
    explanation: string;
  }
  ```
- **UI:** слева — растущий промпт (новые куски подсвечиваются), справа — «ответ AI», который меняется при каждом шаге. Снизу — оставшиеся улучшения как «ступени лестницы».
- **Проверка (детерминированная, без LLM):** сравниваем собранный набор элементов с `optimalSet`. Частичный балл = доля совпадения. «Ответ AI» полностью заготовлен и выбирается по набору применённых элементов — никакой генерации.
- **Почему это хороший кастомный тип:** показывает не *что* правильно, а *почему* — даёт пользователю прожить эффект каждого элемента промпта.

**Запасные идеи кастомного компонента** (если захочется заменить): `TokenVisualizer` (интерактивная разбивка текста на токены + ползунок лимита контекста, показывающий, что «обрезается») или `ConfidenceCalibration` (отвечаешь + оцениваешь уверенность; портал показывает калибровку — учит не доверять Am/себе вслепую).

## Визуальное различение типов

Каждый тип имеет свою **иконку и акцентный цвет** в шапке `ExerciseShell` (через CSS-переменную `--exercise-accent`), чтобы пользователь сразу понимал формат: выбор / соединение / порядок / ввод / поиск / сборка. Иконки — из одного набора (напр. `lucide-react`) для единообразия.
